# Advanced Flow

With flexible visual configuration, you can use various blocks to build data flow logic. This helps you quickly create automation flows for different scenarios. It also makes it easy to achieve smart control of IoT devices.

## Create an Automation

To learn how to create an advanced flow, please see [Automation - Create an Automation](./../automation.mdx#create-an-automation).

Then, you can [add](#how-to-add-blocks) and connect different [blocks](#blocks) on the canvas. Build device interaction logic for your specific needs.

## Blocks

In Aqara Studio, blocks serve as the "information hub" and "control center" of automation. They transform complex backend automation logic (such as "turn on the light when someone is present and the room is dark") into clear, visual device states and one-click scene switches, making invisible automation intuitive and controllable.

### How to Add Blocks

1. Click the **+** button in the left sidebar of the canvas.
2. Browse and select the desired block type (such as Trigger, Condition, or Action).
3. Under the desired block type, further select a specific block. For example, if you click **Time** under the **Trigger** type, you will see the **Periodic Trigger**, **Scheduled Trigger**, and **Loop Trigger** blocks.
4. Click the desired block and the block will appear on the canvas.

### Components

Each block can consists of the following elements:

| Component | Description |
| -- | -- |
| Input point | If a block has an input point, it can be triggered.          |
| Parameter  | Used to set the state to monitor, the action to perform, or other settings. |
| Output point | If a block has an output point, it can trigger other blocks.   |

### Categories
Advanced Flow supports the following types of blocks, each serving a specific role:

| Category | Description |
| -------- | ----------- |
| [Trigger](./trigger.mdx)   | The **starting node** of an Advanced Flow.<br />Triggers can be initiated by various conditions, such as device status changes (e.g., a light being turned on), scheduled times, or alarm events, serving as the entry point for the flow. |
| [Condition](./condition.mdx) | The **intermediate node** of an Advanced Flow.<br />Used to determine whether device states, time, or various logical/comparison operations meet the preset conditions, and thereby control the direction of the flow. |
| [Action](./action.mdx) | The **ending node** of an Advanced Flow.<br />Used for sending and executing various actions, including controlling single or multiple devices, data processing, delays, etc., to achieve a closed automation loop. |

### Connection

The connection methods for different [block categories](#categories) are as follows:

- [Trigger](./trigger.mdx): It only has output points, no input points. The output of a trigger block can connect to multiple other blocks, acting as the starting point of a flow.
- [Condition](./condition.mdx): It can have both input and output points. Outputs from multiple blocks can connect to the input of a condition block; each output of the condition block (such as different judgment results) can also connect to multiple subsequent blocks, enabling branching in the flow.
- [Action](./action.mdx): As the end point of the flow, it supports multiple input points, and can be triggered by several blocks at once. Depending on actual needs, multiple output points can also be set to further extend the flow.

:::tip
The input and output points of Flow blocks are primarily used for flow control. For example, multiple inputs mean that flows from different blocks can converge on a single block; multiple outputs mean that the flow from one block can branch out to several subsequent steps.

These inputs and outputs are not related to the transfer of real "data"; all data processing and computation are handled internally within each block, and are not passed directly between blocks.

Additionally, the outputs of [Condition blocks](./condition.mdx) provide both true/false branches to further control the direction of the flow.
:::
