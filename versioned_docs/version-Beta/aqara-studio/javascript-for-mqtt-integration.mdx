# JavaScript Script for MQTT Integration

This document introduces how to use a JavaScript script in Aqara Studio to add device data points and their parsing methods.

## Script Overview

The script **must** include the following 4 JavaScript methods:
- [parseDiscoveryResponse](#parsediscoveryresponse)
- [parseDiscoverPointsResponse](#parsediscoverpointsresponse)
- [parseReportRequest](#parsereportrequest)
- [buildWriteRequest](#buildwriterequest)

:::tip
Do not modify the parameters or the input and output structures of these methods.
:::

## Console Logging

Scripts can output diagnostic logs through `console.log`, `console.info`, `console.debug`, `console.warn`, and `console.error`. Multiple arguments are joined with spaces, while `warn` and `error` messages use the `[warn]` and `[error]` prefixes. Logs appear in the read-only `Console Content` field on the protocol configuration page and are limited to 2048 characters; older content is replaced when the limit is exceeded. Each script parser accepts up to 20 log entries per second; excess entries are discarded and summarized in the next time window. Each entry retains up to 512 characters and converts only the first 16 arguments. Do not log unconditionally inside loops or high-frequency message handlers; use a debug condition or sampled logging instead.

## parseDiscoveryResponse

This method declares the MQTT devices to be added to Aqara Studio.

### Device Object Structure

| Field Name | Description |
|-----------|-------------|
| deviceName | Device name. Optional; an empty value is treated as an empty string. |
| deviceId | Device ID. Required. |
| brokerEndpoint | MQTT Broker address. Required. |
| brokerPort | MQTT Broker port. Required; the default port is `1883`. |
| connectionType | Connection type. Required; the default value is `3`. Supported values: <ul><li>0: Anonymous</li><li>1: SSL anonymous</li><li>2: SSL username/password</li><li>3: Username/password without SSL</li></ul> |
| username | Username. Required depending on `connectionType`. |
| password | Password. Required depending on `connectionType`. |

### Example

```javascript
function parseDiscoveryResponse() {
	return [
		{
			deviceName: "TestDevice",
			deviceId: "98CC4D189E60",
			brokerEndpoint: "127.0.0.1",
			brokerPort: 1883,
			connectionType: 3,
			username: "admin",
			password: "public"
		}
	];
}
```
## parseDiscoverPointsResponse

This method declares to Aqara Studio "which data points this device supports". Aqara Studio passes `deviceId` as the first parameter when calling this function.

You need to return an array containing one or more data point objects (such as `pointId`, etc.; for more details, please refer to [Point Object Structure](#point-object-structure)). Aqara Studio will call this function during the discovery process of data points, parse the return value, and thus identify all data points supported by the device, enabling seamless integration and control of these points.

### Point Object Structure

| Field Name | Description |
|-----------|-------------|
| pointId   | Data point ID. Please consult the device manufacturer to obtain the actual ID to allow Aqara Studio to discover this point. |
| pointName | Name for the data point defined by you. |
| devType | Corresponds to `deviceType` in the Aqara Data Model Specification. For the specific available values, please refer to [Device Types](../aqara-developer/aqara-data-model-specification/device-types.mdx). |
| functionCode | Corresponds to `functionCode` in the Aqara Data Model Specification. For the specific available values, please refer to [Function Codes](../aqara-developer/aqara-data-model-specification/function-codes.mdx). |
| traitCode | Corresponds to `traitCode` in the Aqara Data Model Specification. For the specific available values, please refer to [Trait Codes](../aqara-developer/aqara-data-model-specification/trait-codes.mdx). |
| dataType  | Data point type. Supported values: <ul><li><code>bool</code>: Boolean</li><li><code>numerical</code>: Numeric</li><li><code>enum</code>: Enumeration</li><li><code>text</code>: Text</li></ul> |
| enumRange | Enumeration options. Required only when a general data point uses `dataType: "enum"`. The array must not be empty. Each item contains an integer `value` and a string `label`; both must be unique, and `label` must not be empty. |
| access    | Access capability bit mask. <ul><li>1: Readable</li><li>2: Writable</li><li>3: Readable and writable</li></ul> |
| subTopic  | (**Only applicable for MQTT devices**) Aqara Studio subscribes to this topic to get the latest data for this point. Please fill in according to your device protocol or manufacturer documentation. |
| pubTopic  | (**Only applicable for MQTT devices**) Aqara Studio sends control message data to this topic for the data point. Please fill in according to your device protocol or manufacturer documentation. (For read-only points, leave this field empty) |

:::tip
When adding new data points, you may choose one of the following approaches depending on your needs:

- **Based on Aqara Spec**: You need to fill in fields such as `devType`, `functionCode`, and `traitCode`, which enables the data point to be compatible with the Aqara Spec.
- **General Approach**: You only need to fill in the basic fields such as `dataType` and `access`.
:::

### Example

```javascript
function parseDiscoverPointsResponse(deviceId) {
	return [
		// Add points based on Aqara Spec
		{
			pointId: "1#Chiller Operation Status",
			pointName: "1#Chiller Operation Status",
			devType: "AirConditioner",
			functionCode: "Output",
			traitCode: "OnOff",
			subTopic: "/WEI-iEdge/realdata/xxx",
			pubTopic: "/WEI-iEdge/write/req/xxx"
		},
		// Add points by the general approach
		{
			pointId: "1#Chiller On/Off Read",
			pointName: "1#Chiller On/Off Read",
			dataType: "bool",
			access: 1,
			subTopic: "/WEI-iEdge/realdata/xxx",
			pubTopic: "/WEI-iEdge/write/req/xxx"
		},
		{
			pointId: "1#Chiller Operating Mode",
			pointName: "1#Chiller Operating Mode",
			dataType: "enum",
			enumRange: [
				{ value: 0, label: "Off" },
				{ value: 1, label: "Cooling" }
			],
			access: 3,
			subTopic: "/WEI-iEdge/realdata/xxx",
			pubTopic: "/WEI-iEdge/write/req/xxx"
		},
		{
			pointId: "1#Chiller Outlet Water Temp Set Read",
			pointName: "1#Chiller Outlet Water Temp Set Read",
			dataType: "numerical",
			access: 2,
			subTopic: "/WEI-iEdge/realdata/xxx",
			pubTopic: "/WEI-iEdge/write/req/xxx"
		},
		{
			pointId: "1#Chiller Cooling Capacity Percentage",
			pointName: "1#Chiller Cooling Capacity Percentage",
			dataType: "numerical",
			access: 2,
			subTopic: "/WEI-iEdge/realdata/xxx",
			pubTopic: "/WEI-iEdge/write/req/xxx"
		}
	];
}
```

## parseReportRequest

This method extracts the current value of a specified data point (`pointId`) from the reported data sent by the device.

When Aqara Studio receives a message reported by the device, it will call this function and pass in the data point ID and the original message string. You need to implement logic in this function to extract the value corresponding to the specified `pointId` from the `values` field. In this way, Aqara Studio can update and display the value of the data point in real time.

:::tip
Please customize the parsing logic based on the actual data reporting protocol of your device to ensure you can accurately extract the numeric value corresponding to `pointId` from the original message (for example, `obj["values"][pointId]` in the example).
:::

### Example

```javascript
// The following example parses a JSON message string reported from an MQTT device
function parseReportRequest(pointId, jsonStr) {
	try {
		const obj = JSON.parse(jsonStr);
		return obj["values"][pointId]?.toString();
	} catch (e) {
		return null;
	}
}
```

## buildWriteRequest

This method constructs the command string to control the device based on the data point ID to control and the value to set.

When Aqara Studio needs to send a control command to a device (such as switch on or off the device, set temperature), it will call this function and pass in the device ID, target data point ID, and value to be set. The function must return a string that meets the control command format required by the device protocol, usually including device identifier, timestamp, and specific value fields.

Please construct the object structure within the function according to the actual acceptable message format of your device. For example, for an MQTT device, you may include fields like `node`, `group`, `timestamp`, and write the value corresponding to `pointId` in the `values` field.

### Example

```javascript
// Example: Control an MQTT device via a JSON string
// The value parameter is automatically passed in by Aqara Studio; no need to manually declare or assign it
function buildWriteRequest(deviceId, pointId, value) {
	try {
		const obj = {
			"node": "Chiller",
			"group": "1#Chiller",
			"timestamp": 1768204054654,
			"values": {

			},
			"errors": {

			},
			"metas": {

			}
		};
		obj.values[pointId] = parseFloat(value);
		return JSON.stringify(obj);
	} catch (e) {
		return null;
	}
}
```
